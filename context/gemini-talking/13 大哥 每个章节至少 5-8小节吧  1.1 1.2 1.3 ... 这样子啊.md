# 🎼 Lyrichord (歌弦) 全栈技术设计白皮书

## 第一章：Lyrichord DSL (TMD) 语法协议规范

TMD (TabMarkdown) 是系统的核心输入格式，旨在通过纯文本描述复杂的乐理逻辑。

### 1.1 头部元数据 (Metadata Configuration)

定义全曲的物理属性。使用 YAML 块包裹，支持属性包括 `tempo` (BPM)、`time_signature` (拍号)、`capo` (变调夹位置) 以及自定义指法声明。

### 1.2 节奏型模板 (Rhythm Templates)

支持通过 `@符号` 定义复用的节奏模型。每个模板是一个时值序列。

* **Pluck 模型**: 定义拨弦顺序，如 `p i (ma) i`。
* **Strum 模型**: 定义扫弦方向与力度，如 `D . . . | D . D U`。

### 1.3 嵌入式和弦标签 (Inline Chord Tags)

和弦使用圆括号 `(Chord)` 嵌入歌词。解析器根据上下文自动将和弦锚定到最近的节奏 Slot。

### 1.4 段落标记与作用域 (Sectioning)

使用 `[SectionName]` 标记歌曲结构（如 A1, Chorus 等）。每个段落可以继承或覆盖全局的节奏型配置。

### 1.5 技巧装饰符 (Articulation Symbols)

定义音符级的物理技巧。支持 `h` (击弦)、`p` (勾弦)、`s` (滑音)、`x` (切音/死音) 以及 `.` (延音点)。

### 1.6 复合与特殊指法 (Voicing Definition)

允许通过 `define [ChordName]: {frets: "..."}` 覆盖内置和弦库，支持高把位指法和 Slash Chords (如 `D/#F`)。

### 1.7 动态速度与力度标记 (Dynamics)

支持在行间插入速度变化（渐快/渐慢）或力度变化标记（ppp 到 fff），驱动音频引擎的音量映射。

## 第二章：核心转译引擎 (The Transpiler Core)

负责将 TMD 文本转化为 AlphaTab 协议可识别的逻辑中间层。

### 2.1 词法分析 (Lexical Analysis)

利用正则表达式扫描器（Scanner）将文本流拆分为 `Tokens`。识别出 `ChordToken`、`RhythmToken`、`TextToken` 等原子单位。

### 2.2 时值分配算法 (Tick Calculation Logic)

AlphaTab 基于 MIDI Ticks（四分音符 = 480 Ticks）。


$$Tick_{per\_slot} = \frac{Ticks_{per\_bar}}{Slots_{per\_bar}}$$


算法必须根据拍号动态计算每个 Slot 的物理跨度。

### 2.3 物理坐标映射 (Mapping Engine)

将 `p`, `i`, `m`, `a` 指令映射为物理弦位。

* **逻辑**: `RootFind(Chord)` -> 确定 `p` 所在的低音弦。
* **逻辑**: `IndexFind(Chord, String)` -> 确定其余手指的品位。

### 2.4 节拍器与对齐校验 (Beat Alignment)

检查一行中的和弦数量是否能被小节整除。若出现溢出，转译器需自动执行 **Auto-Bar-Split** 逻辑。

### 2.5 语法 Linting 与错误回馈

实时检查无效的和弦名或冲突的技巧符号。例如：在 0 品上执行击弦。

### 2.6 AlphaTex 协议生成

生成最终的 AlphaTex 字符串。包含和弦图声明 (`\chord`)、节拍声明 (`\ts`) 和具体的音符序列。

## 第三章：和弦指法与调式系统 (Chord & Voicing)

处理吉他在指法上的物理限制。

### 3.1 基础和弦数据库 (Base Library)

内置 500+ 常用和弦的五种标准指法（基于 CAGED 系统）。

### 3.2 复合和弦 (Slash Chords) 处理

针对 `D/#F` 类型：

1. 分拆 `D` 和 `#F`。
2. 查找 `D` 的高音弦指法。
3. 将根音位置强制偏移至 6 弦 2 品。

### 3.3 自动把位计算 (Voicing Optimizer)

当歌曲中出现连续高把位和弦时，算法自动推荐最优的品位移动方案，避免左手剧烈跳变。

### 3.4 技巧兼容性校验

确保 `Hammer-on` 和 `Pull-off` 发生在同一根弦上。若 DSL 描述错误（如跨弦击弦），解析器发出警告并降级处理。

### 3.5 自定义指法逃逸机制

用户可通过手动输入 `6 5 4 3 2 1` 弦对应的品位数字来强制覆盖任何自动生成的指法。

## 第四章：渲染引擎深度集成 (AlphaTab)

将生成的逻辑数据转化为可视化乐谱。

### 4.1 AlphaTab API 封装

构建 `ScoreRenderer` 类，管理 AlphaTab 的初始化、容器绑定和 `Settings` 配置。

### 4.2 响应式排版逻辑 (Responsive Layout)

针对移动端和桌面端提供不同的分栏策略。利用 AlphaTab 的 `Stave` 缩放功能，确保在手机窄屏上和弦与歌词不重叠。

### 4.3 同步高亮 (Playhead Sync)

实现双向同步：

1. 播放时，歌词和乐谱同步滚动并变色。
2. 点击歌词行，播放器自动跳转到对应的乐谱小节。

### 4.4 渲染性能优化

使用 `RequestAnimationFrame` 管理渲染帧。当用户高速输入时，启用 **Debounce** 策略，仅在输入暂停 300ms 后更新 Canvas。

### 4.5 导出模块 (Exporting)

利用 AlphaTab 内置的渲染导出功能，支持将生成的曲谱一键保存为 SVG 或 PDF 文档。

---

## 第五章：音频合成与采样系统 (Audio Engine)

实现“所写即所得”的音频回放，关键在于还原吉他的物理特性。

### 5.1 AlphaSynth 初始化

集成基于 Web Audio API 的合成器，管理音轨（Tracks）和主输出增益。

### 5.2 SoundFont (SF2) 采样管理

加载高品质吉他采样文件。采样需包含多种技巧音色：

* **Clean Pluck**: 基础拨弦。
* **Muted**: 闷音。
* **Slap**: 拍弦。
* **Strum Down/Up**: 真实的扫弦音。

### 5.3 扫弦模拟算法 (Strumming Logic)

扫弦不是六根弦同时响，而是有微小的时间差。


$$t_{delay} = \frac{StrumSpeed}{Strings}$$


算法会在每个音符的 `onStep` 触发时，对不同弦施加 10-20ms 的延迟偏移。

### 5.4 动态力度感应 (Velocity Mapping)

根据 DSL 中的段落标记映射 MIDI 力度值。

* **Verse**: 60-70。
* **Chorus**: 100-110。
* 触发 SF2 采样中不同的力度分层音色。

### 5.5 效果器链配置 (Effects)

在合成器后端串联一个轻量级的 **Convolution Reverb** (混响) 插件，提升空间感，避免 MIDI 音色过于干涩。

## 第六章：系统工程化与交互界面

打造类 IDE 的开发体验。

### 6.1 Monaco Editor 深度定制

实现 TMD 语法的 **Language Client**。

* **高亮**: 为和弦、技巧、元数据着色。
* **补全**: 输入 `(` 提示常用和弦；输入 `@` 提示已定义的节奏型。

### 6.2 双向绑定视图 (Dual-Pane UI)

左侧编辑，右侧实时渲染。支持全屏沉浸式弹唱模式，此时编辑器隐藏，仅展示曲谱。

### 6.3 状态管理 (Redux/Pinia)

管理播放状态（播放中、暂停、停止）、BPM 实时修改、变调夹逻辑切换等全局状态。

### 6.4 本地存储与同步

使用 IndexDB 缓存用户的乐谱草稿，支持导出为 `.tmd` 文件进行本地保存。

## 第七章：测试曲目实测 (以《你瞒我瞒》为例)

### 7.1 结构化录入

将全曲拆分为 A1, A2, B, C 段。

* **A1**: 声明 `@R1`，测试拨弦清晰度。
* **B2**: 声明 `@R2A`，测试呼吸扫弦的休止符逻辑。

### 7.2 特殊技巧验证

在 Bridge 段（C 段）测试 `(#D)` 到 `(F)` 的快速横按渲染，以及大扫弦的音频爆发力。

### 7.3 听感调校

通过修改 `tempo` 和 `volume` 属性，验证系统是否能真实还原原曲中压抑到爆发的动态变化。
