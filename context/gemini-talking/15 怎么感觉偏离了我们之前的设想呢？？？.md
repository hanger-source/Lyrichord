理解了，握个手。这种“偏离感”通常来自文档写得太像教科书，而少了那种**“工程师在白板前对着架构图指点江山”**的实战感。

我们的核心设想没变：**做一个最高效的缝合怪，把文本、和弦、节奏一下子变成“能看又能听”的动态谱。** 既然要“深”，我们不玩虚的。下面我直接按照你要求的 **1.1、1.2、1.3...** 这种地毯式排查的逻辑，把这个系统的**每一个转轴和齿轮**是怎么咬合的，写得清清楚楚。

# 🎼 Lyrichord (歌弦) 系统架构深度设计（实战版）

## 第一章：DSL 文本协议层 (The Input Standard)

我们要解决如何用最少的键盘输入，描述出最复杂的吉他动作。

### 1.1 全局上下文解析 (Global Context)

系统启动时，解析器首要任务是建立“时空背景”。比如 `time_signature: 4/4`。这不只是一个显示标签，它决定了后续所有 Tick 计算的**分母**。如果是 `6/8` 拍，解析器会自动将原本四分音符的脉冲宽度缩短为原来的 75%。此外，`capo` 变调夹的设定会直接进入“坐标偏移栈”，当用户写下 `(G)` 时，系统底层渲染出的品位会自动减去 Capo 的数值，但音频引擎发出的频率维持原调，保证“所见即所弹”。

### 1.2 节奏模板的“槽位”分拆逻辑 (Slot Decomposition)

这是你的核心创新点。一个 `@Rhythm` 模板不再是一串死代码。解析引擎会将 `@R1: p-3-(12)-3` 拆解为一个拥有 4 个 **Atomic Slots (原子槽位)** 的数组。每个槽位不仅承载音符，还附带了“时值权重”。如果用户输入 `@R1(fast)`，解析器会自动应用一个时值缩放因子（Scaling Factor），在不改变 BPM 的情况下，让这几个音符在谱面上靠得更近，从而实现“即兴变速”的听感。

### 1.3 和弦锚点映射算法 (The Anchoring Algorithm)

为了实现“一下子导入”，系统必须处理杂乱的文本。解析器采用**“正则捕获 + 索引定位”**。它会扫描歌词行，记下每个 `(Chord)` 出现的 Character Index。随后，它将歌词行的总物理长度映射到一小节的 Tick 总数上。例如：歌词全长 20 个字，和弦出现在第 10 个字，解析器会精准地将该和弦挂载在第 960 Tick（半小节处）。这解决了用户无需手动对齐，只要把和弦塞进歌词里，系统就能自动排版。

### 1.4 技巧符号的二进制转换 (Technique Flags)

吉他的物理技巧（击勾、切音、扫弦）在 DSL 中被定义为 **Technique Flags**。当扫描到 `h` (Hammer-on) 时，解析器会在抽象语法树中为该音符打上 `0x01` 的标记。转译到 AlphaTex 时，这个标记会触发 `L` 指令；而在音频合成层，它会触发 SF2 采样中的“无拨片触发”层。这种多层映射机制，让一个简单的字母 `h` 具备了视觉和听觉的双重生命力。

## 第二章：转译引擎层 (The Logic Transpiler)

这是你说的“缝合”工作的核心，负责把 DSL 翻译成 AlphaTab 能懂的“外语”。

### 2.1 音乐抽象语法树 (Music-AST) 的构建

不同于代码编译器，我们的 AST (抽象语法树) 是**带有时间维度的**。树的根节点是曲目，分支是小节（Bar），叶子是音符（Note）。每个 Note 对象必须包含：`String` (弦号)、`Fret` (品位)、`Duration` (时值) 和 `Velocity` (力度)。有了这棵树，我们才能实现“自动播放”：播放头只需要根据时间轴遍历这棵树，就能像自动钢琴一样驱动 AlphaSynth 发声。

### 2.2 Tick 高精度对齐引擎 (Tick Precision Engine)

由于 Web 浏览器存在时钟漂移，我们必须在底层使用 **Tick 绝对坐标**。系统将全曲定义为一个线性增长的 Tick 轴（四分音符 = 480）。转译时，引擎会执行一个 `Bar-Fill` 算法：如果一个小节没填满（比如用户只写了一个和弦），引擎会自动补齐休止符（Rest），防止后续小节错位。这是保证《你瞒我瞒》那种精准节奏感的技术底线。

### 2.3 自动分小节与换行策略 (Line-Breaker Logic)

吉他谱最怕一行挤太满。转译引擎内置了一个**“视觉压力传感器”算法**。它会计算每一行 AlphaTex 产生的音符密度，如果密度超过阈值（通常是一行 4 个小节），它会自动插入 `\break` 指令强制换行。同时，它会计算歌词的宽度，确保歌词不会因为和弦太密而重叠，这在目前市面的开源谱站中是非常罕见的高级特性。

## 第三章：音频合成与采样系统 (The Sound Engine)

解决“自动播放”的问题，让你知道拨弦和扫弦具体的听感。

### 3.1 基于 SF2 的采样分层加载 (Sample Layering)

为了让声音不像 MIDI，我们要加载带 **Velocity Layers** 的采样。比如，当你轻拨弦时，SF2 触发的是采样点 A；当你重扫弦时，SF2 切换到带高频谐波的采样点 B。我们的音频引擎会根据 DSL 里的段落标签（如 `[Chorus]`）自动调大力度偏移量（Velocity Offset），让副歌听起来自带“气势爆发”。

### 3.2 扫弦的物理延时模拟 (Strum Timing Simulation)

这是提升“听感”的绝招。真正的扫弦，六根弦发声有约 20ms 的时差。

我们的转译器在处理 `@Strum` 时，不会把六个音符写在同一个 Tick。它会生成一组带有**微小偏移量**的音符序列。当你听到这段代码回放时，耳朵会捕捉到那种由低向高划过的“唰”的一声，而不是死板的“哐”的一声。

### 3.3 实时混响与音频空间化 (Audio Spatialization)

为了让《你瞒我瞒》前奏那种压抑感更写实，我们在合成器后端挂载了一个 **Web Audio ConvolverNode**（卷积混响）。通过在 DSL 头部设置 `reverb: small_room`，系统会加载一段房间脉冲响应采样。这让吉他声听起来像是在一个小房间里弹奏，而不是直接从电脑主板里蹦出来的电流音。

## 第四章：交互与渲染层 (The User Experience)

解决“快速进入”和“实时反馈”的问题。

### 4.1 Monaco Editor 与 AlphaTab 的双向绑定

我们不只是简单的刷新。系统通过一个 **Virtual DOM 映射表** 记录谱面位置。当你在编辑器第 10 行修改了一个和弦，AlphaTab 只会重绘对应的小节。这种增量渲染方案，保证了即使在配置较低的电脑上，你改谱子时也能获得毫秒级的实时反馈，这才是真正的 IDE 体验。

### 4.2 “录音模式”与音轨回看

既然是平台，你可以开启录音同步。系统通过 WebRTC 录制你的声音，并将其与自动播放的吉他伴奏进行时间轴对齐。当你回看时，你会发现你的歌声和系统生成的乐谱是同步滚动的，这对于检查自己是否抢拍、和弦是否按错极其有用。

### 4.3 导出与分享的协议化 (Serialization)

导入导出不需要传输巨大的文件。整个曲谱被序列化为一个特殊的 **Lyrichord JSON**。这个文件非常小（通常只有几 KB），你可以像粘贴代码一样把它分享给别人。别人打开链接，系统瞬间重新运行转译引擎，乐谱和声音立刻就在对方的浏览器里“复活”了。
