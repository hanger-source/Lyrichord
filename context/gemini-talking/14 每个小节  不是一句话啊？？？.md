# 🎼 Lyrichord (歌弦) 工业级全栈技术设计方案

## 第一章：Lyrichord DSL (TMD) 语法协议深度规范

### 1.1 头部元数据声明 (Metadata Configuration)

元数据块是曲谱的“配置文件”，采用严格的 YAML 格式。它不仅定义了标题、BPM（每分钟节拍数）和拍号（如 4/4, 6/8），还承载了**全局渲染配置**。例如，`capo` 属性不仅是一个数字，它会触发解析引擎对所有和弦品位进行物理位移计算。此外，用户可以在此声明全局的 `velocity`（力度）基准，为后续音频引擎提供动态参考。

### 1.2 节奏型模板递归逻辑 (Rhythm Matrix)

节奏型模板是 DSL 的灵魂，它解决了“文本驱动音符序列”的问题。每一个 `@标记` 定义了一个时值数组，解析器会将其映射到一个固定长度的 **Tick 槽位（Slot）** 中。支持嵌套定义，例如一个 4/4 拍的模板可以包含 8 个八分音符位，每个位置可以放置 `p`（根音）、`i/m/a`（手指代号）或 `(123)`（多弦共奏）。

### 1.3 嵌入式锚点算法 (Chord-to-Lyrics Anchor)

和弦标签 `(Chord)` 的解析采用**“就近原则”与“等分原则”**的混合算法。当一行歌词中出现多个和弦时，解析引擎会计算字符间距，将和弦自动锚定到最接近的拍子起点。如果一行只有一个和弦，它默认占据整个小节；如果有两个，则自动平分为两个二分音符的时值。这种设计保证了用户在不输入复杂时值的情况下，依然能获得准确的节奏对齐。

### 1.4 技巧装饰符系统 (Articulation Syntax)

为了支持吉他特有的物理技巧，DSL 引入了**非侵入式装饰符**。在节奏序列中，`h` 代表击弦（Hammer-on），`p` 代表勾弦（Pull-off），`s` 代表滑音。解析引擎在处理这些符号时，会自动在生成的 AlphaTex 代码中插入 `L`（Legato）连奏指令。此外，`x` 符号会被映射为死音（Dead Note），在谱面上渲染为十字架，并在音频中触发短促的打击采样。

## 第二章：核心转译引擎架构 (The Transpiler Engine)

### 2.1 词法扫描与抽象语法树 (AST Generation)

转译器的第一阶段是利用**正则表达式有限自动机**对 TMD 文本进行全量扫描。它将文本流切分为元数据节点、节奏定义节点和演奏内容块。这些节点会被组装成一棵 **Abstract Music Tree (AMT)**。这棵树不仅包含文本信息，还包含每一个音符在时间轴上的绝对 Tick 位置，为后续的音频同步播放提供唯一真理源。

### 2.2 工业级时值计算模型 (Tick Math Logic)

系统采用 MIDI 标准，将一个四分音符定义为 **480 Ticks**。


$$Tick_{per\_measure} = \frac{480 \times 4 \times Numerator}{Denominator}$$


转译引擎会根据拍号动态计算每小节的总 Tick 数。当用户使用 `@R1` 这种 8 步进节奏时，引擎会自动计算 $480 \times 4 / 8 = 240$ Ticks 每个 Slot。这种高精度的计算模型支持了切分音、三连音等复杂节拍，确保了回放时的节拍稳定性。

### 2.3 物理坐标映射引擎 (Physical Coordinate Mapping)

这是将“乐理符号”转化为“物理指法”的关键步骤。当引擎解析到 `(G) @p` 时，它会启动 **Root-Finder 算法**：首先从和弦库检索 G 指法 `[3, 2, 0, 0, 0, 3]`，然后根据 `p` 指令寻找最低音弦（6 弦 3 品），并输出 AlphaTex 坐标 `3.6`。如果遇到 `123` 这种组合，引擎会并行计算这三根弦的品位，生成组合音符指令。

### 2.4 节拍器对齐与自动补全 (Auto-Bar Balancing)

为了防止用户录入的节拍出现“溢出”或“短缺”，引擎内置了**自动对齐校验器**。如果一行中的和弦总时值不足一小节，引擎会自动插入休止符（Rest）；如果超过一小节，则根据拍号强制执行 **Auto-Bar-Split** 逻辑，将多余的和弦推移至下一小节。这保证了渲染出的乐谱始终符合标准乐理规范，不会出现混乱的排版。

## 第三章：和弦库与指法解析系统 (Chord System)

### 3.1 动态和弦指法库 (Dynamic Chord DB)

系统内置了一个基于 **JSON 结构的工业级指法库**，涵盖了从基础三和弦到复杂的九和弦、十三和弦。每个指法条目不仅包含品位坐标，还包含建议的按弦手指。对于变调夹（Capo）开启的情况，指法库会执行**坐标偏移映射**，确保渲染出的品位数字是相对于 Capo 的位置，而音频引擎触发的是真实的物理频率。

### 3.2 复合和弦与 Slash Chords 处理逻辑

针对 `D/#F` 这种复合和弦，解析引擎采用**双重查找逻辑**。首先解析主和弦 `D` 获取高音弦的指法，然后解析斜杠后的 `#F`。如果 `#F` 是根音，系统会覆盖原有的最低音，强制将根音 Slot 定位在 6 弦 2 品。这种处理方式完美解决了《你瞒我瞒》等华语流行歌中常见的下行 Bass Line 渲染问题。

### 3.3 自定义指法逃逸符机制 (Custom Voicing)

为了应对指弹（Fingerstyle）中可能出现的非标准指法，DSL 支持**逃逸符语法**。用户可以通过 `(MyChord){x 3 2 0 1 0}` 直接定义弦位坐标。解析引擎在遇到大括号时会暂停自动查找逻辑，直接提取数组中的数字进行渲染。这为高级用户提供了极大的灵活性，支持任何奇怪的变调指法。

## 第四章：渲染引擎深度集成 (AlphaTab)

### 4.1 AlphaTab API 封装与状态管理

我们将 AlphaTab 的 Canvas 渲染器封装在 `ScoreRenderer` 类中。该类通过 **Observer 模式** 监听转译引擎输出的 AlphaTex 变化。每当用户在编辑器中修改一个字符，渲染器会执行增量更新。为了保证流畅度，我们使用了 `requestAnimationFrame` 进行渲染帧同步，确保在复杂的全曲谱渲染时，UI 界面依然保持 60fps 的响应速度。

### 4.2 响应式乐谱布局策略 (Responsive Layout)

乐谱渲染最难的是移动端适配。我们利用 AlphaTab 的 `Stave`（五线谱/六线谱）缩放逻辑，实现了一套**动态列宽算法**。在桌面端，系统会采用双栏展示以利用屏幕宽度；在移动端，系统自动切换为单栏垂直滚动模式，并增大和弦图的显示尺寸，确保用户在手机弹唱时能清晰看清每一个品位点。

### 4.3 歌词与谱面实时同步逻辑 (Lyric-Sync)

这是一个极其注重体验的功能。我们通过计算每一个音符在 AlphaTab 内部的坐标，建立了一个 **Coordinate Map**。当播放头移动到特定 Tick 时，系统会高亮对应的歌词文字。反之，用户在预览区点击某一行歌词，系统会自动调用 AlphaTab 的 `api.scrollToMeasure()` 方法，将谱面精准滚动到对应的位置，实现沉浸式交互。

## 第五章：音频合成与真实感回放 (Audio Engine)

### 5.1 基于 Web Audio 的合成器管道 (Synth Pipeline)

音频模块集成了 **AlphaSynth**，这是一个高性能的软采样器。我们为其构建了一个多级的 Audio Node 链条，包括 `MasterGain`（主音量控制）、`ConvolutionReverb`（卷积混响）和 `Compressor`（压缩器）。这种设计模拟了真实的吉他效果器链，使得最终输出的声音不再是干巴巴的 MIDI 音，而具有一定的空间感和动态。

### 5.2 SoundFont (SF2) 采样分层技术

回放的真实感来源于采样。系统加载的 **Acoustic Guitar SF2** 文件包含了多个力度层。当转译引擎在副歌部分（如 B 段）输出较高的 Velocity 值时，音频引擎会自动触发“重拨”采样，产生更清脆的琴弦振动声。此外，SF2 库中还包含了扫弦（Strum）的专门采样，用于模拟手指划过琴弦时的物理延迟感。

### 5.3 扫弦物理延迟算法 (Strum Timing Emulation)

真正的扫弦不是 6 根弦同时发声。我们实现了一个**“下行/上行延迟模型”**：


$$t_{string\_n} = t_{start} + (n \times \Delta delay)$$


其中 $\Delta delay$ 根据扫弦的速度动态调整（通常为 10ms-25ms）。这种微小的毫秒级时差，能够让音频引擎模拟出真实的拨片划过感，彻底告别死板的电子合成音听感。

## 第六章：工程化与用户体验设计

### 6.1 Monaco Editor 深度定制 (IDE Experience)

作为一款开源工具，编辑器体验至关重要。我们为 TMD 语法编写了专门的 **Language Definition**，支持：

* **语法高亮**：和弦显示为橙色，元数据为紫色，技巧标记为蓝色。
* **智能提示**：当用户输入 `@` 时，自动弹出已定义的节奏模板列表；输入 `(` 时，智能推荐常用和弦名，极大提升录入速度。

### 6.2 实时预览与离线存储 (State Management)

前端采用 **Vite** 进行热构建，并使用 **IndexedDB** 存储用户的本地草稿。这意味着即便用户刷新浏览器或断网，之前编写的《你瞒我瞒》曲谱也不会丢失。同时，我们通过 **Web Worker** 将繁重的转译运算（DSL to AlphaTex）移出主线程，确保用户在进行长篇录入时，页面打字响应依然丝滑。
